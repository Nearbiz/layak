3.3 The credential format (this is the standard)
The signed Layak credential is the most strategically important artifact in the demo. It is what makes Layak a standard, not an app. Format:
{
  "iss": "tng.layak.my",
  "sub": "worker_kumar_001",
  "iat": 1735200000,
  "exp": 1735203600,
  "jti": "lyk_2026_a3f9c1...",
  "credential": {
    "score": 720,
    "tier": "excellent",
    "earnings_12mo": 112400,
    "earnings_6mo_monthly_avg": 4280,
    "consistency": 0.91,
    "income_sources": 4,
    "tenure_months": 18,
    "trade": "barber_services",
    "zone": "brickfields_kl"
  },
  "consent": {
    "scope": ["earnings", "score", "tenure"],
    "lender_id": "aeon_credit",
    "expires_at": 1735203600
  }
}
.<HMAC-SHA256 signature>
DECISION: HMAC, not RSA
Real Verifiable Credentials use asymmetric crypto. We don't. HMAC-SHA256 with a shared secret is 10 lines of Go and demos identically. The judges care that there IS a signature, not which algorithm. We will name-drop "v2 will move to W3C VC + Ed25519" in Q&A.

5. The Layak Score (the formula on a slide)
The score is the most-questioned artifact in Q&A. It must be defensible in one breath: "Five signals, weighted, calibrated against the collective." If Aein cannot recite the formula from memory, the demo is not ready.
5.1 Signal definitions
Signal
Weight
Formula
Interpretation
Consistency
30%
1 - (stddev / mean) of monthly earnings over 6 months
Stable income → trustworthy income
Tenure
20%
log(months_active + 1) / log(24), capped at 1.0
Longer track record → lower risk
Diversity
20%
min(unique_counterparties / 30, 1.0)
Many customers → not dependent on one
Volume
20%
min(total_6mo_earnings / 30000, 1.0)
Real income scale, calibrated to MYR
Recency
10%
txns_last_30d / txns_30d_avg
Active workers → currently earning
5.2 Final score mapping
raw = 0.30*consistency + 0.20*tenure + 0.20*diversity + 0.20*volume + 0.10*recency
score = 300 + round(raw * 550)        // maps [0,1] → [300, 850]

tier = excellent  if score >= 740
       good       if score >= 670
       fair       if score >= 580
       building   otherwise
5.3 Kumar's worked example (memorize this)
Kumar's 6-month data:
  monthly earnings: [4100, 4280, 4150, 4350, 4220, 4280]   mean=4230, std=92
  consistency = 1 - (92/4230)         = 0.978
  tenure (18mo) = log(19)/log(24)      = 0.926
  diversity (62 customers / 30)         = 1.000  (capped)
  volume (25,380 / 30,000)              = 0.846
  recency (32 txns / 28 avg)            = 1.143  → cap at 1.0

raw = 0.30(0.978) + 0.20(0.926) + 0.20(1.0) + 0.20(0.846) + 0.10(1.0)
    = 0.293 + 0.185 + 0.200 + 0.169 + 0.100
    = 0.947

score = 300 + round(0.947 * 550) = 300 + 521 = 821
TUNING NOTE
Kumar's tuned score should land at 720 in the demo, not 821. Adjust seed earnings volatility OR the diversity cap to land him there. The deck and screenshots assume 720. Aein: get this right by hour 6.

ERROR HANDLING — 24HR RULE
Every endpoint returns 200 or hardcoded 4xx. No retry logic. No circuit breakers. No rate limits. If the demo fails, we restart the server. This is fine.
